Creating the design library

Background:

As part of redesign and transformation of the platform, the design team had to decide what to do with the library. Basically, there were two options: create a new one or reuse the old one.

The team

As part of the city’s online portal redesign project, the design team had to decide what to do with the library. Basically, there were two options: create a new one or reuse the old one.
Reuse an existing library or set up a new one? Originally there was an old library with two linked documents with 700+ screens inside. The idea was to simply change the styles and appearance of the components, but the library was not ready for this due to several nuances.
  1. Fundamentally new design During the redesign of the platform, all layouts, elements and user experience have undergone major changes. (Unlike, for example, my other project, where we specifically redesigned to keep the HTML structure and design system.)
  2. Mockups Assembly Even though we wouldn’t have had to build the layouts from scratch, they would still have had to be radically adjusted to look right, so it wouldn’t have saved much time.
  3. Detached components Quite common practice in old files, so even if we updated the components, we would not be able to run the changes.
  4. Poorly assembled components A lot of components were poorly assembled, were not responsive, and had useless memory-cluttering layers.
  5. Unused components There were a lot of components made «just in case» (for example, variations for a dark theme, although there was no dark theme at the moment).
  6. Hard to navigate structure All components and styles were displayed on one page, so it was difficult to find what you needed. Moreover, the components were not grouped into variants (at that time this feature was new).

Final decision

In view of all these problems, we decided to create a new library because redesigning old components took as much time (or even more) than creating new ones (taking into account the problems that were in the outdated library). It was a chance to start over and avoid previous mistakes.

Design Library Research

As soon as the decision to create a new library was made, we started our research with benchmarking. I looked at over 20 different design systems and libraries, the main aspects we wanted are:
  • The structure of the file and pages, how the components are grouped.
  • How components are assembled.
  • How components are decomposed in terms of the atomic approach.
We also interviewed some other designers in our company who have dealt with extensive project libraries. Not only did they share some useful knowledge in terms of structuring, but also in terms of useful tools and techniques.
Design Library Structure We knew for sure that the option with all components on one page would not suit us, because it is difficult to navigate, so we decided to use different pages. There was a question of choosing a grouping of components. After going through a lot of design systems, we settled on a structure grouped by type: banners, cards, buttons, etc. But we needed to group not only components, we also had styles, design tokens and guidelines. This is how two main sections of our library appeared: Foundations and Components. We also decided to flag pages to show the status of being ready. If the component is completed, it is marked in green, if it is in progress and not completed, it is marked in yellow.
Atomic Design The atomic design approach was used in the legacy library, but we’ve taken it to the next level with the new one. Combined with variants, our library became very flexible. Now we were able to design organisms like filter panels, where the smallest atom was a checkbox.

Workflow Changes

Library refactoring wouldn’t be that effective without changing the workflow. Here are some changes that made designers’ life easier.
  1. Evolutionary Development. In order not to waste time on components that may never be used, we decided to develop our library in an evolutionary way, and create only those components that are needed.
  2. Component quality control. There is a page called «To Review» where the designer posts a new component to be added to the library. I, as responsible for the library and review, checked the assembly of the component, the necessary states and descriptions.
  3. No detaching. Strict component usage policy. In the final design review phase, we make sure that components from the library are used, otherwise missing components are added to the library.
  4. Specifications. Enhanced practices with specifications for developers.

  • Poor assembly: useless layers and lack of responsiveness.
  • No variants, set of components with a long name.

The process of adding components and reviewing

New services and screens were made by a large number of designers from different pools. As a result, the following problems could appear:
  1. Duplicate components. Sometimes a designer can create a new component even though a similar one may already exist, and it is more logical to add a new state to it.
  2. The naming of a new component can be poorly chosen.
  3. A component may be assembled incorrectly, contain empty layers or layers with the wrong naming, be non-adaptive and not have all states.
  4. A component may be assembled incorrectly, contain empty layers or layers with the wrong naming, be non-adaptive and not have all states.
  5. A component may be placed in a library in such a way that it may be difficult to find it.
Therefore, it was decided to conduct a review of the components. When a new component appeared, the designer had to show it on a meeting, and if this was not possible, then place it in the To Review tab. Once in a while, I or a colleague checked this tab, commented or made changes, after which the component was migrated to its corresponding page.

Results

  1. Proper assembly of components greatly speeds up the design work. Designers don’t have to tweak components or reinvent the wheel because components are responsive and behave as expected in layouts.
  2. Variations is a huge plus, as designers don’t have to look through the entire library to find what they need.
  3. The new library structure has made navigation easier. It has become not only easier to find suitable components for layouts, but also to minimize the error of creating duplicate components.
  4. Getting rid of unnecessary layers and states has speeded up work in Figma, as files take up less space.
  5. The developers appreciated the new library. They noted the usability, the quality of the description of the guides, as well as all the states of the components. Even though we didn’t have a full-fledged design system, it made it more convenient for developers to work with layouts, and reduced refinement and communication time.
  6. The design library assumed a redaction policy for articles and events, which contributed to the preparation of well-formed articles by content managers.
The design system was redesigned as part of the redesign of City.Online portal.
